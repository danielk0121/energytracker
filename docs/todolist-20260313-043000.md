# 에너지트래커 기술 고도화 상세 보고서 (Senior Engineer View)

본 문서는 **에너지트래커** 프로젝트의 Phase 1 프로토타입 분석 결과를 바탕으로, 클린 코드와 성능 최적화, 그리고 Phase 2(실제 OCR 및 클라우드 연동)를 위한 구체적인 작업 목록을 정리한 보고서입니다.

## 🧐 현재 상태 분석 및 개선 방향
- **코드 구조**: `index.html`이 950라인을 넘어서며 CSS, JS, HTML이 강하게 결합되어 있어 유지보수가 어려움. -> **모듈화 필요**
- **데이터 처리**: `IndexedDB` 도입으로 저장 공간은 확보했으나, 대량 데이터 렌더링 시 $O(N^2)$ 복잡도의 로직이 산재함. -> **알고리즘 최적화 필요**
- **기능 완성도**: 사진 분석 로직이 Mock 함수 기반임. -> **Tesseract.js 등 실엔진 연동 필요**

- [ ] 001 **CSS/JS 분리**: `index.html` 내의 스타일과 스크립트를 외부 파일로 분리하여 코드 가독성 및 캐싱 효율 증대.
- [ ] 002 **차트 렌더링 모듈화**: `renderDailyChart`, `renderMonthlyChart` 등 캔버스 또는 DOM 기반의 차트 로직을 클래스나 모듈로 캡슐화.
- [ ] 003 **TypeScript 도입**: 복잡한 UI 상태 관리와 데이터 구조에 대한 타입 안정성을 위해 현대적인 개발 환경으로 마이그레이션.

## 2. 성능 최적화 (Performance Optimization)
- [ ] 004 **목록 렌더링 알고리즘 개선**: `renderUsageList`에서 `findIndex`를 사용하여 데이터를 찾는 로직($O(N^2)$)을 Map 객체 캐싱 기반($O(N)$)으로 변경.
- [ ] 005 **이미지 리사이징 최적화**: `IndexedDB`에 저장하기 전, 분석에 불필요한 고해상도 이미지를 브라우저 사이드에서 리사이징하여 저장 공간 및 메모리 최적화.
- [ ] 006 **가상 렌더링(Virtual Scrolling) 검토**: 수백 개 이상의 상세 사용 기록 목록을 렌더링할 때 브라우저 부하를 줄이기 위해 화면 노출 영역만 렌더링.
