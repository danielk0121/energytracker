# 에너지트래커 기술 고도화 보고서 (Senior Engineer View)

본 문서는 **에너지트래커** 프로젝트의 Phase 1 프로토타입 분석 결과를 바탕으로, 시스템의 안정성, 확장성, 성능 최적화를 위한 향후 작업 목록을 정리한 보고서입니다.

## 🧐 현재 상태 분석
- **아키텍처**: 단일 HTML/JS 파일 구조로 초기 진입 장벽은 낮으나, 비즈니스 로직과 UI 렌더링 코드가 강하게 결합되어 있음 (약 900라인).
- **데이터 관리**: `sessionStorage`를 활용한 상태 보존은 훌륭하나, 이미지 데이터를 문자열로 저장하는 방식은 브라우저 할당량(5MB) 문제로 실서비스 적용 시 크래시 유발 가능성이 큼.
- **성능**: 데이터 변경 시마다 전체 DOM을 재생성하는 구조로, 26개월 이상의 대량 데이터 처리 시 반응 속도 저하 예상.

## 🚀 향후 작업 TODO List

### 1. 시스템 안정성 및 저장소 최적화 (Priority: High)
- [ ] **IndexedDB 도입**: 사진 데이터를 `sessionStorage`에서 `IndexedDB`로 이전. (수 GB 단위의 대용량 이미지 데이터 안정적 관리 가능)
- [ ] **에러 핸들링 고도화**: 파일 업로드 용량 제한 체크, 이미지 분석 실패 시 재시도 로직 및 사용자 피드백 강화.
- [ ] **상태 관리 객체화**: 산재된 전역 변수들을 하나의 전역 `State` 객체로 통합하고, 변경 시에만 렌더링이 트리거되도록 구조화.

### 2. 코드 품질 및 유지보수성 (Refactoring)
- [ ] **모듈화 (ES Modules)**: `index.html` 내의 CSS, 데이터 로직, 차트 로직, UI 이벤트를 별도 파일로 분리.
- [ ] **React/TypeScript 전환**: Phase 2의 OCR 엔진 연동 및 복잡한 UI 상태 관리를 위해 현대적인 프레임워크로 마이그레이션 준비.
- [ ] **중복 로직 공통화**: 월별/일자별 차트의 공통 렌더링 로직(`drawYAxisAndGrid`)을 클래스 또는 함수형 유틸리티로 재사용 가능하게 수정.

### 3. 성능 최적화 (Performance)
- [ ] **가상 렌더링(Virtual Scrolling) 검토**: 상세 기록 목록이 수백 행 이상으로 늘어날 경우, 화면에 보이는 부분만 렌더링하도록 최적화.
- [ ] **DOM 조작 최소화**: `innerHTML` 전체 교체 대신, 변경된 데이터 행(Row)만 `appendChild` 또는 `insertRow`로 추가하도록 수정.
- [ ] **복잡도 개선**: 목록 렌더링 시 중첩 루프 내부에서 호출되는 `findIndex` 로직을 Map 객체 기반의 캐싱 기법으로 변경 ($O(N^2) \to O(N)$).

### 4. Phase 2 (OCR) 기술 검토
- [ ] **Tesseract.js 프로토타입**: 실제 라이브러리를 연동하여 월패드 사진에서 숫자 및 날짜 패턴을 추출하는 POC(Proof of Concept) 수행.
- [ ] **데이터 보정 알고리즘**: OCR 오인식(예: 8과 0의 혼동)을 방지하기 위해 이전/이후 데이터의 흐름을 분석하는 보정 로직 설계.

### 5. 아키텍처 확장 (Cloud)
- [ ] **개인 클라우드 연동 설계**: README의 방향성에 따라 Google Drive API 연동 인터페이스 설계. 중앙 서버 없이 사용자 개인 저장소에 데이터가 백업되는 구조 구현.
